MISS CLAUDE — MULTI-SESSION WORKTREE ROLES & GUARDRAILS


PURPOSE

The operator runs many Claude Code sessions at once and is NOT a git expert. These guardrails exist
to keep the operator from making mistakes, not to teach git. Every session: speak plain English, keep
git detail short, lead with status, and end with the one safe next step.

The short version lives in CLAUDE.md ("Roles & branch workflow"). This file is the full reference.


WHAT THIS REPO IS

  Miss Claude (a.k.a. the Mission Dashboard) — a dependency-free, Python-3-stdlib web UI served
  by two systemd units:
    mission-dashboard.service   the HTTP app (app.py, port 4200)
    claude-console.service      the ttyd Claude-console bridge (port 4201)
  "Application code" = app.py, the console-*.sh / install.sh scripts, the *.service units, and
  anything under scripts/. There is no build step; "deploy" means restart the service (below).


PLAIN LABELS

  feature worker   a normal coding session, in its own worktree on a claude/<slug> branch
  integrator       the one session that accepts finished branches into staging
  staging          the `working` branch (integration branch)
  deploy branch    the `master` branch (release; what the live services run from)


BRANCH MODEL

  working (staging) --ff--> master (deploy branch) --deploy--> live service restart

  Feature workers branch off `working` into ~/missclaude-worktrees/<slug> on claude/<slug>.
  The integrator fast-forwards finished claude/<slug> branches into `working`.
  Promoting `working` -> `master` (release) and deploying are separate, separately-approved steps.


WHAT "DEPLOY" MEANS HERE

  No build, no remote push to a prod box. Deploy = make the running services load the committed code:
    sudo systemctl restart mission-dashboard.service     # app.py changes (port 4200)
  console-session.sh / console-launch.sh changes are picked up by the NEXT console session opened
  (no restart needed); restart claude-console.service only to change ttyd itself, and note that
  restarting it ends live console sessions. Deploy is INTEGRATOR-ONLY, after YES DEPLOY (the hook
  blocks sudo/systemctl for feature workers).


HOW A SESSION KNOWS ITS ROLE

  Launch wrappers export CLAUDE_MISS_ROLE:
    scripts/claude-miss             -> feature   (creates/enters a worktree; FEATURE WORKER banner)
    scripts/claude-miss-integrator  -> integrator(runs in the primary checkout on `working`)
  If CLAUDE_MISS_ROLE is unset, assume feature worker (the restrictive default).


SESSION-START STATUS (every session opens with this, then waits)

  STATUS:
  GREEN/YELLOW/RED — one sentence (role, clean/dirty, current-with-working / behind).

  WHAT CHANGED:
  Plain-English summary (skip on a fresh session).

  SAFE NEXT STEP:
  Exactly what to tell which Claude next.

  NEEDS APPROVAL?
  The exact phrase needed, if any.

  Stoplight:
    GREEN  — safe to continue.
    YELLOW — needs an approval phrase before continuing.
    RED    — stop and ask the integrator / operator.


APPROVAL PHRASES (operator must type EXACTLY; nothing vaguer counts)

  YES COMMIT         feature worker may commit its own changes
  YES REBASE         feature worker may rebase its branch onto current `working`
  YES INTEGRATE      integrator may fast-forward a reviewed feature branch into `working`
  YES PUSH WORKING   integrator may push `working` to origin (if a remote is ever added)
  YES RELEASE        integrator may move `master` forward to `working`
  YES DEPLOY         integrator may restart the service to load the released code

  "ok", "do it", "sounds good", "continue", "fix it" are NOT enough. Ask for the exact phrase.
  These phrases are enforced by Claude's behaviour — the hook cannot read the chat.


FEATURE WORKER — rules

  May:
    - edit project code in ITS OWN worktree
    - run app.py locally / lint / syntax-check / other checks
    - show a plain-English summary
    - say "ready for integrator"
    - after YES COMMIT: git add by explicit path + git commit (never `git add .` / `-A`)
    - after YES REBASE: rebase its branch onto current `working`
  Must not:
    - update `working` (staging)
    - touch `master` (deploy branch)
    - push
    - deploy / restart services (sudo systemctl)
    - edit or clean up other worktrees
    - make git changes outside its own branch


INTEGRATOR — rules

  Does not write feature code. Reviews and integrates finished branches.
  Before integrating, confirm: branch is clean, based on current `working`, changed files are expected.
  May:
    - review feature branches (diff, log, status)
    - after YES INTEGRATE: git merge --ff-only <claude/branch>   (fast-forward only)
    - after YES PUSH WORKING: git push origin working   (if a remote exists)
    - after YES RELEASE: move `master` forward to `working`
    - after YES DEPLOY: restart the service (sudo systemctl restart mission-dashboard.service)
  Must not:
    - write feature code (*.py, *.sh, *.service, scripts/)
    - force-push (ever)
    - do non-fast-forward merges

  If a branch will not fast-forward, it is behind `working`. Do NOT rebase it yourself — tell the
  operator to have that branch's feature worker rebase it (YES REBASE), then re-check.


HARD ENFORCEMENT (the hook)

  .claude/hooks/prevent-misswork.py is a PreToolUse hook and the ONLY hard guardrail, because the
  wrappers launch Claude with --dangerously-skip-permissions (the permission allowlist is bypassed).
  It reads CLAUDE_MISS_ROLE and hard-blocks (exit 2) the forbidden actions:

    main/master (any role): the full dangerous set (commit/push/add/reset/rebase/merge/restart/
                            pip/rm/mv/cp/chmod/chown/sudo/redirects/...).
    feature worker:         push, merge, systemctl(deploy), git worktree, branch delete/rename,
                            switching away from its branch, sudo; and edits to other worktrees or the
                            primary checkout. (git add/commit/rebase, running app.py, edits in its own
                            worktree, and /tmp writes stay allowed — the YES phrases gate the rest.)
    integrator:             editing application code (*.py *.sh *.service scripts/), force-push,
                            non-ff merge, rebase. (docs/ and .claude/ edits are allowed.)

  The hook is a backstop. The role discipline and approval phrases are still your responsibility.


LAUNCH WRAPPERS (scripts/, run from the repo)

  scripts/claude-miss              feature worker. Inside a worktree -> enter & launch. Elsewhere ->
                                   prompt for a task name, create ~/missclaude-worktrees/<slug>
                                   + claude/<slug> off working, then launch. Prints the FEATURE WORKER
                                   banner + stoplight status.
  scripts/claude-miss-integrator   integrator. Runs in ~/mission-dashboard on `working`;
                                   refuses elsewhere. Prints the INTEGRATOR banner and working status.

  Preview either without starting a session:  CLAUDE_MISS_DRYRUN=1 scripts/claude-miss
